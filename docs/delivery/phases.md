# Delivery Phases

## Purpose
Define the phased delivery model for turning this repository from planning into a usable product.

## Intended Audience
Founder, delivery lead, and future implementation agents sequencing work.

## Phase 0: Planning Baseline
- Establish repo guidance, architecture truth, delivery docs, and Codex operating rules.
- Exit when the repository can support structured implementation planning without ambiguity.

## Phase 1: Foundations
- Prepare the first implementation plan for source ingestion, rubric configuration, editorial workflow states, publishing flow, and analytics instrumentation.
- Exit when the team can start coding with clear architecture, scope, and validation criteria.

## Phase 2: Internal MVP Build
- Implement the one-owner, one-channel, Telegram-only workflow.
- Exit when the owner can run the core content loop with real data and review controls.

## Phase 3: Stabilization
- Improve reliability, latency, duplicate suppression, and editorial quality signals.
- Exit when the workflow is trusted for daily operations.

## Phase 4: Expansion Readiness
- Prepare for multi-channel support, richer role models, and selective automation depth.
- Exit when the system can evolve without reworking the core domain.

## Key Decisions
- Planning is its own phase and must be treated as real delivery work.
- Implementation should not begin until Phase 1 produces a concrete build plan.

## Open Questions
- Should Telegram source ingestion land in Foundations or later within the MVP build?
- Which milestone should gate any limited auto-publish experiment?

## Next Actions
- Turn phase exits into milestones and acceptance criteria.
