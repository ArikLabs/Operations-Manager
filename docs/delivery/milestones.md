# Milestones

## Purpose
Define concrete milestones that future implementation sessions can target.

## Intended Audience
Founder, delivery lead, and future implementation agents.

## M0: Repo Ready For Implementation Planning
- Root guidance exists.
- Product, architecture, delivery, and Codex docs exist.
- Skills scaffolding and MCP strategy exist.

## M1: Build Plan Approved
- A formal implementation plan exists for the first coding phase.
- Workstreams, acceptance criteria, and dependencies are explicit.
- Open architecture questions are reduced to manageable decisions.

## M2: Source-To-Draft Spine Designed
- Source, event, rubric, and draft flows are specified tightly enough for implementation.
- Safety and analytics requirements are linked to the workflow.

## M3: Editorial And Publish Workflow Designed
- Review states, approval rules, scheduling, and publish semantics are fully specified.
- Operational recovery and audit expectations are explicit.

## M4: MVP Ready For Internal Trial
- The first usable workflow supports one owner and one Telegram channel.
- Success metrics can be measured from real usage.

## M5: MVP Trusted For Daily Use
- Latency, approval rate, and publish reliability meet baseline expectations.
- The owner can rely on the workflow without reverting to fully manual operations.

## Key Decisions
- Milestones should represent operational readiness, not only completion of tasks.
- M1 is the gate to writing serious product code.

## Open Questions
- Which source connectors must be part of the first internal trial?
- What quantitative thresholds should define "trusted for daily use"?
